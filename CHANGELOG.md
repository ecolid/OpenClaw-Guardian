# OpenClaw Guardian 更新日志

## [v1.4.6] - 2026-03-07
### 视觉与监控大改版 (Visual Dashboard Overhaul)
- **`/status` 面板拟态化重构**: 抛弃了简陋的纯文本信息，引入了类似 Windows 任务管理器的终端极客画风面板。
- **硬件指引系统**: 利用 Python 算法动态渲染 `[██████░░░░]` ASCII 进度面板，并辅以 `🟢🟡🔴` 交通灯红黄绿情绪刻度，让服务器负载状态一目了然。
- **高价值致死参数捕获**: 在内存和硬盘之外，新增了存活决定性因素 `Swap 交换分区` 用量的捕获，以及 CPU 当前峰值占比的单独析出。
- **双重时间锚点**: 现在不但提供 `宿主机 1/5/15m 负载` 和 `宿主机启动时间 (Uptime)`，还精确展示了 `OpenClaw 引擎` 守护存活了多少个小时。

---## [v1.4.5] - 2026-03-07
### 紧急修复 (Hotfix)
- **启动级消息列队阻断 (Boot Queue Flush)**: 深度修复了“死亡循环”。此前由于用户在死机或死循环期间反复点击按钮，导致 Telegram 积压了巨量且未确认的 `/update_rollback` 消息。在此更新中，我在守护程序的 `main()` 初始化阶段植入了一张“清道夫网络”：机器人在启动时会先抛弃并清空 Telegram 发信服务器上所有积压的历史遗留弹药。从此，重启必干净，永不断触，永不死循环。

---## [v1.4.4] - 2026-03-07
### 紧急修复 (Hotfix)
- **`/update_rollback` 无限死亡循环**: 修复了点击恢复上一个版本会导致机器人无限自动重启的问题。此 Bug 的起因是由于底层 `systemctl restart openclaw-guardian` 触发过快（毫秒级），在切断 Python 进程前，Telegram 并未来的及确认收到该指令的 Ack 回执。这就导致无论机器人重启多少次，开机后的第一件事永远是从未读列表重抓 `/update_rollback` 并再次自杀。现已在重启前加入 `time.sleep(3)` 延时器，为 Telegram 确认消息的处理留出时间。

---## [v1.4.3] - 2026-03-07
### 新特性 & 体验升级 (New Features & Polish)
- **底层报错消息转义 (HTML Escape)**: 彻底解决了由于日志出现 `<400>` 等尖括号导致被 Telegram 识别为非法 HTML 标签而引起的“静默被吞”现象。现在所有 grep 查询的上下文内容都会得到安全的 HTML 实体化处理，保证 100% 送达你的手机屏幕。
- **状态面板重构 (`/status`)**: 去除了冰冷的占位符，新增了运维界最具价值的两个黄金指标：**服务器已运行时长 (Uptime)** 和 **系统 1/5/15 分钟负载 (Load Average)**。此外，监控兵接管的每日问候语中会显式声明当前的 `VERSION` 型号。
- **防鬼手误触锁 (Rollback Confirm)**: 对 `/rollback` 历史快照恢复机制进行了二次确认分层。现在点击某一条备份时不再是“一触即死”的即时覆盖，而是会弹出一道拥有明确 `⚠️ 危险操作确认` 和 `取消` 选择权的红区隔离网。

---## [v1.4.2] - 2026-03-07
### 紧急修复 (Hotfix)
- **核心语法树逃逸修复**: 抱歉，这是由于我的代码替换工具本身存在的 JSON 转义缺陷导致的“幽灵回滚”。上一版本的修复代码在提交到 GitHub 时，因为正则表达式解析错误，并没有真正被替换掉！这导致了哪怕你拉取了 v1.4.1，底层依然在执行错误的 `bash` 提取导致 `ash` 报错。本次更新已通过系统级底层全量强行覆写，真正锁死了 `\$0` 全局变量，现在所有检索必能百分百拉取到 5 条精准历史。

---## [v1.4.1] - 2026-03-07
### 修复 & 优化 (Fix & Enhancement)
- 真正的最终版变量转义修复：在合并 `v1.4.0` 的高可用代码时，我不慎将原本已经修好的 `\$0` 脱逸语句回滚成了 `\\$0`！这导致 Bash 在二次加载热更新时再次将环境变量污染为 `bash`（提取后报错缺少 `b` 字母成为 `ash`）。现已在 `v1.4.1` 重新上锁焊死。
- 自动更新轮询周期从 30 分钟 (1800s) 极致压缩至 **10 分钟 (600s)**。这是受限于 GitHub Raw CDN 缓存能力的物理极限，逼近真实意义上的“实时推送”。

---## [v1.4.0] - 2026-03-07
### 新特性 (New Features)
- **OTA 热更新探针**: 引入自动化后台轮询线程，每 30 分钟嗅探一次 GitHub main 分支。当发现新版本修复和特型放出时，Bot 机器人主动向主阵地推送更新提示（连按键都送到了手里）。
- **Grep 排他锁 (Mutex)**: 全局阻断连点按钮导致的并发搜索任务。只要前一个 `DataInspectionFailed` 还在搜，再次点击必定拒接防卡机，从底层保供 VPS 算力。
- **僵尸任务熔断超时**: Python 核心现已被加入 `timeout=60` 强行阻断沙盒。如果底层检索耗时越过 60 秒红线，进程即时被处决，向控制端回复超时警报，永绝无响应挂起后患。

---## [v1.3.5] - 2026-03-07
### 修复 & 优化 (Fix & Enhancement)
- 彻底根治了多层嵌套语言（Bash -> Python -> Bash）转义时的极端死角 Bug（引致 awk 提示 invalid char）。现在底层引擎稳如磐石，`journalctl -r` 的时光倒流检索机制全面生效。
- 采纳用户意见平衡检索极限，将历史抛出数量由 `3` 次扩大到最高 **`5`** 次，最大化利用 Telegram 单人气泡消息的物理容量，确保每次探针都饱满且不断连。

---## [v1.3.3] - 2026-03-07
### 优化 (Enhancement)
- 重构 `/grep` 的内核检索引擎。抛弃原来的 “固定查最后 800 行” 的浅层策略，改为 **全局倒序检索 (Reverse Search)**。
- 引入智能防卡死机制：从最新往历史倒翻日志，只要集齐**最近的 3 次案发现场**就会强行终止检索。即使报错发生在前天，也能在几秒钟内给你捞出来，再也不会有“找不到记录”的空炮指令。

---## [v1.3.2] - 2026-03-07
### 修复 (Fix)
- 根治了由于 `cut` 截断管道导致的 Linux 底层 `SIGPIPE` 问题。改用更安全的 `awk` 截断逻辑，确保在大规模日志提取期间 Telegram 查询请求不会异常中断或吞掉消息。

---## [v1.3.1] - 2026-03-07
### 修复 (Fix)
- 优化 `/grep` 和 `/logs` 指令的底层系统开销。引入字符级 `cut` 截断（强制每行最多读取500字符），防止 OpenClaw 极其庞大的超长 JSON 把系统内存卡爆并导致 Telegram 回复断连。保护提取现场稳定输出。

---## [v1.3.0] - 2026-03-07
### 新增 (New)
- 改进 `/grep` 指令交互：单独发送 `/grep` 时，将呼出一个“快捷诊断面板”(Inline Keyboard)。内置了防打扰的一键回查按钮：
  - [🔴 百炼风控拦截 (DataInspectionFailed)]
  - [🟡 网络请求超时 (Timeout)]
  - [💥 内存耗尽被杀 (Out of memory)]
  - [🚨 严重运行异常 (Exception/Error)]
  - [🔵 系统启动记录 (Started)]
- 支持查询被风控拦截的关键词 (例如 `/grep 400`)，并将案发现场前后 5 行的 Payload 上下文推送到 Telegram，避免了原始 `/logs` 的刷屏与线索遗漏问题。

---## [v1.1.0] - 2026-03-07
### 新增 (New)
- 支持 OTA 热备份机制，通过 `/update` 命令无感平滑升级。
- 支持安全回滚机制，通过 `/update_rollback` 一键恢复上一版本。
- 新增版本控制与更新日志播报系统。

### 优化 (Enhancement)
- 分离私有配置到本地 `.env` 文件，不再硬编码敏感 Token。
- 增加防抖热备脚本拦截，避免在高速编辑文件时频繁触发全量备份。

### 修复 (Fix)
- 在执行备份打包 (tar) 时过滤 `.png`, `.jpg`, `.mp4` 等大体积的多媒体文件，防止磁盘 IO 和小龙虾 Playwright 缓存撑爆节点崩溃。
